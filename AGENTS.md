# Attendance - руководство для агентов

Это основной контекстный файл для coding agents в этом репозитории. Перед началом работы прочитай его и не создавай отдельные `content.md`, `context.md` или `server.md`, если пользователь явно этого не попросил.

## Цель проекта
- Attendance - сервис для учета посещаемости и отметок об отсутствии учеников.
- Основной сценарий: быстро выбрать класс, ученика и добавить период отсутствия с причиной и комментарием.
- Репозиторий должен оставаться компактным и понятным.
- Текущий стиль архитектуры: небольшой Node.js/Express монолит.

## Продуктовые приоритеты
- Быстрые действия пользователя.
- Минимум шагов до результата.
- Один понятный рабочий экран без лишних промежуточных страниц.
- Простая поддержка: пользователь не должен разбираться в сложной структуре проекта.

## UX-инварианты
- Не увеличивай количество страниц без явной необходимости.
- Главный пользовательский поток держи на `/attendance`: выбор класса, список учеников, выбранный ученик, форма новой отметки и история.
- Для небольших изменений предпочитай доработку текущего workspace вместо добавления новых экранов.
- Если меняется UI или пользовательский поток, проверяй критичный сценарий в браузере/Playwright, когда это применимо.
- В шаблонах не добавляй лишние объясняющие блоки; интерфейс должен работать без инструкции на экране.

## Технологии и структура
- Backend/UI: Node.js ESM, Express, Handlebars.
- Точка входа: `attendacs.js`.
- Шаблоны: `views`.
- Основной layout: `views/layouts/main.hbs`.
- Стили сейчас подключаются через Tailwind CDN в layout.
- Работа с MySQL вынесена в `vendor/db.mjs`.
- Логи вынесены в `vendor/logs.mjs`.
- Runtime-логи пишет код в `vendor/logs`; эти файлы не должны попадать в Git.
- Express сейчас раздает статику из папки `public`. В репозитории есть `publick`, поэтому перед добавлением ассетов сначала проверь фактическое имя папки и не плодить дубли.

## Перед каждой задачей
- Прочитай этот `AGENTS.md`.
- Проверь текущий `git status` и не откатывай чужие незавершенные изменения.
- Держи scope минимальным и безопасным.
- Предпочитай существующие паттерны и локальные helper API.
- Не добавляй новые зависимости, папки, страницы или абстракции без явной пользы.
- При работе с БД сначала изучи `vendor/db.mjs` и текущие переменные окружения.
- Учитывай, что часть строк в старых файлах могла иметь проблемы с кодировкой; не ухудшай русские тексты.

## Команды
- Установка зависимостей: `npm install`.
- Локальный запуск: `npm run start`.
- Dev-запуск с nodemon: `npm run dev`.
- Прямой запуск: `node attendacs.js`.
- Тесты: `npm test` сейчас является placeholder-командой и завершится ошибкой, пока реальные тесты не добавлены.
- Порт по умолчанию: `789`, можно переопределить через `PORT`.

## Переменные окружения
- Локальные секреты храни только в `.env` или другом gitignored-файле.
- Не коммить реальные значения `.env`.
- Используемые переменные:
  - `PORT`
  - `MDBHOST`
  - `DBUSER`
  - `DBPASS`
  - `DBNAMESUSR`
  - `SSO_BASE_URL`
  - `SSO_CLIENT_ID`
  - `SSO_CLIENT_SECRET`
  - `SSO_CALLBACK_URL`
  - `SSO_SERVICE_ID`
  - `JWT_SECRET`
  - `AUTH_SESSION_SECRET`
  - `AUTH_SESSION_COOKIE_NAME`
  - `AUTH_SESSION_TTL`
  - `AUTH_DISABLED` только для локальной разработки.
- В production `SSO_CLIENT_SECRET`, `JWT_SECRET` и `AUTH_SESSION_SECRET` должны задаваться через окружение PM2 или `.env`, а не храниться в Git.

## Границы данных
- SSO и общие школьные справочники являются внешним read-only контуром.
- Сервис SSO для Attendance: `sso.srvs.name = atten`, текущий `SSO_SERVICE_ID=13`.
- Вход выполняется через `https://platoniks.ru/sso`; callback production: `https://stud.platoniks.ru/api/cb`.
- Роли доступа бери из JWT claim `right`, выбирая максимальный `role_id` только для `srv_id == SSO_SERVICE_ID`.
- Классы бери из `sso.kaf_name` с `type = 1`; учеников бери из `sso.users` с `type = 1` и `status = 1`.
- Не создавай локальные таблицы классов/учеников, если эти данные уже есть в SSO.
- Не пиши в `sso.*`, `school_local.sso_users`, `school_local.sso_kafs` и не меняй SSO-схему миграциями.
- Не выполняй destructive SQL-операции без прямого запроса и понятного rollback-плана.
- Не меняй схему БД наугад: сначала найди источник данных и согласуй миграционный подход.
- Не печатай содержимое `.env` и реальные credential values в ответах.

## Сервер и деплой
- Серверный путь проекта: `/var/www/html/attendance`.
- Production-доступы и секретные команды хранятся в локальном `AGENTS.local.md`; этот файл намеренно не должен попадать в Git.
- Перед деплоем всегда проверяй фактическую схему запуска на сервере: `pm2`, `systemd`, cron или ручной Node-процесс.
- Не трогай production без необходимости.
- После деплоя проверяй процесс, HTTP-ответ главного сценария и свежие runtime-логи.
- Если нужен неинтерактивный SSH-сеанс, сначала проверь PATH/версию Node на сервере.

## Git-дисциплина
- Держи изменения атомарными и сфокусированными.
- Не коммить секреты, приватные ключи, токены, `.env`, runtime-логи и локальные агентские файлы.
- Работай с существующими пользовательскими изменениями, не откатывай несвязанные правки.
- Перед финальным отчетом проверь `git status --short`.
- Если файл с секретами уже оказался tracked, не удаляй его молча: сообщи пользователю и предложи безопасный план.

## Отчеты
- Не создавай дополнительные отчетные markdown-файлы без запроса пользователя.
- В финальном ответе кратко указывай, что изменено, что проверено, и какие риски остались.
- Если доступен локальный `AGENTS.local.md` с notification-командой, в конце завершенной задачи можно использовать ее согласно локальному протоколу.

## Безопасность
- Новые секреты не добавляй в tracked-файлы.
- Операционные credential values используй только для этого проекта и только когда они реально нужны.
- Не вставляй секреты в логи, issue, commit message, markdown-документацию, которая попадет в Git, или финальный ответ.
